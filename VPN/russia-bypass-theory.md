## 1. Как устроены «белые списки» в России

С весны 2025 года в России появился и стал базовой архитектурой новый режим блокировок — **«белые списки»**. Суть проста: разрешён доступ только к заранее согласованному перечню ресурсов, всё остальное блокируется.

**Что находится в белом списке (декабрь 2025 – март 2026):**
- Госпорталы: Госуслуги, mos.ru, МВД, МЧС, ЦБ РФ
- Соцсети: ВКонтакте, Одноклассники, Max (VK), Дзен
- Видео: Кинопоиск, VK Видео, RUTUBE, Okko, PREMIER
- Банки и финансы: крупнейшие российские банки
- Маркетплейсы: Wildberries, Ozon, Яндекс.Маркет
- Яндекс-сервисы: Яндекс.Облако, music.yandex.ru и др.
- Личные кабинеты операторов: МТС, Билайн, МегаФон, Теле2

**Что за пределами списка** (требует обхода):
- Telegram, YouTube, WhatsApp, Instagram*, Google, Zoom
- Зарубежные СМИ, образовательные платформы
- GitHub, большинство CDN Cloudflare

> ⚠️ Белые списки **применяются преимущественно в мобильных сетях** при шатдауне. Домашние проводные провайдеры (кроме крупнейших) в большинстве случаев DPI по белым спискам не применяют.

## 2. Три модели блокировки

Понимание модели — ключ к выбору метода обхода.

### Модель 1 — IP-блокировка (L3)

Разрешены только конкретные IP-диапазоны (CIDR). Пакеты к «чужим» IP просто дропаются.

**Обход:** VPS на хостинге, чьи IP-диапазоны входят в белый список. Пример: Яндекс.Облако, VK Cloud, DiNet — их CIDR разрешены. Трафик до такого VPS проходит, а уже от него идёт куда угодно.

### Модель 2 — DPI + белый список протоколов

ТСПУ пропускает только «известные» протоколы: HTTP, HTTPS (TLS с распознанным fingerprint), DNS, и т.д. Трафик, похожий на «случайный поток байт», блокируется.

**Обход:** маскировка под легитимный TLS — VLESS+Reality, xHTTP.

### Модель 3 — DPI + белый список доменов/SNI

ТСПУ читает SNI в TLS Client Hello и сравнивает с IP назначения. Если SNI говорит `ya.ru`, а IP указывает на хостинг в Амстердаме — сессия разрывается.

**Обход:** Self-Steal (свой домен, свой сертификат, SNI = реальный домен VPS), либо VPS в российском разрешённом облаке.

## 3. Как работает ТСПУ и DPI

**ТСПУ** — Технические Средства Противодействия Угрозам. Физические устройства, установленные операторами связи по требованию Роскомнадзора на ключевых точках сети.

**DPI** (Deep Packet Inspection) — анализ пакетов «в глубину»:

1. Система пропускает TCP-рукопожатие (SYN/ACK проходят).
2. На этапе **TLS Client Hello** — читается поле SNI.
3. SNI сопоставляется с IP назначения через собственный DNS.
4. Если IP и SNI не соответствуют — соединение разрывается.
5. Дополнительно анализируется TLS fingerprint (JA3/JA4) — по нему видно, что это за «браузер».

**Что умеет ТСПУ к 2026 году:**
- Детектировать классический WireGuard по паттернам UDP
- Распознавать голый OpenVPN по UDP (по заголовкам пакетов)
- Определять Shadowsocks без обфускации
- Сравнивать SNI с реальным IP (SNI-spoofing виден)
- Детектировать VLESS+Reality на некоторых операторах (МТС, МегаФон) по поведенческим паттернам
- Блокировать Cloudflare Tunnel и CDN Cloudflare (таймаутом, не явным RST)

## 5. Протоколы: что работает, что нет

### ✅ Работает надёжно

**VLESS + XTLS-Reality** (при правильной конфигурации)
- Трафик неотличим от HTTPS
- TLS fingerprint можно настроить как Chrome
- Главная уязвимость: несовпадение SNI и IP (решается Self-Steal или российским VPS)

**VLESS + xHTTP транспорт**
- Новый транспорт Xray, имитирует HTTP API-запросы
- Режим `packet-up` дробит трафик, эффективно обманывает DPI
- Лучший вариант для iOS/Android на 2026 год

**AmneziaWG (AWG)**
- Модифицированный WireGuard с обфускацией UDP
- Не переживёт белый список CIDR, если IP сервера не в разрешённой подсети
- Хорош для домашних провайдеров без строгой фильтрации

**Каскад: RU-VPS → EU-VPS**
- Первый хоп — VPS в Яндекс.Облаке/VK Cloud (IP в белом списке)
- Второй хоп — любой зарубежный сервер
- Самая надёжная схема для мобильных сетей

### ⚠️ Работает нестабильно

**Shadowsocks (без обфускации)** — детектируется у части операторов

**Trojan** — медленный, паттерны известны ТСПУ

**gRPC транспорт** — с ноября 2025 вызывает деградацию у МТС/МегаФон

### ❌ Не работает в условиях белых списков

**Классический WireGuard** — UDP-паттерны известны ТСПУ

**OpenVPN по UDP** — детектируется у крупных операторов (по UDP заголовкам); по TCP работает лучше, но медленнее

**Tor Bridges / Snowflake** — трафик выглядит как random bytes → не проходит белый список протоколов

**Cloudflare Tunnel** — заблокирован таймаутом

## 6. Хостинг и выбор VPS: критически важно

Выбор хостера — **решающий фактор** при белых списках по CIDR.

### VPS, чьи IP-диапазоны входят в белый список мобильных операторов:

| Хостер | Статус | Примечание |
|--------|--------|------------|
| **Яндекс.Облако** | ✅ В белом списке | Дороже, но проходит везде |
| **VK Cloud** | ✅ В белом списке | Хорошая цена, стабильно |
| **DiNet** | ✅ Частично | Используется как входной шлюз |
| **h2.nexus** | ✅ IP числятся как RU | YouTube без рекламы |
| **RocketCloud** | ✅ Работает | Популярен в сообществе |
| **Aeza** | ⚠️ Нестабильно | Некоторые подсети в блоке |
| **Hetzner/Contabo/OVH** | ❌ Не проходит | IP за рубежом, не в списке |

> **Лайфхак:** Перед покупкой VPS проверяй IP на предмет попадания в белые списки. Ищи актуальные CIDR-листы в репозитории [allow-domains](https://github.com/itdoginfo/allow-domains/) и в TG-каналах по теме.

## 7. 3x-ui: ключевые параметры профилей для обхода белых списков

### 7.1 VLESS + Reality — базовая конфигурация

Это стандарт де-факто. Ниже — параметры, критические для обхода.

**Inbound настройки:**

| Параметр | Значение | Почему |
|----------|----------|--------|
| Protocol | VLESS | — |
| Port | **443** | Строго 443 — имитация HTTPS |
| Security | **Reality** | Не TLS, а именно Reality |
| Flow | `xtls-rprx-vision` | Обязателен для Reality |
| uTLS | **chrome** | Маскировка под Chrome |
| Dest | `домен-донор:443` | Куда «переадресуем» чужих |
| SNI | тот же домен-донор | Должен совпадать с Dest |

**Требования к домену-донору (Dest/SNI):**
- Не заблокирован РКН
- Поддерживает TLSv1.3 и HTTP/2
- Иностранный сервер (вне РФ)
- Главная страница не делает redirect на другой домен
- IP домена-донора должен совпадать с IP вашего VPS (если используете Self-Steal — см. раздел 8)

**Хорошие примеры доменов-доноров:**
- `addons.mozilla.org`
- `www.swift.com`
- `dl.google.com`

> ❌ **Не используй** `google.com`, `vk.com` как Dest при зарубежном VPS — ТСПУ видит несовпадение IP и блокирует.

### 7.2 VLESS + xHTTP — рекомендуется для 2026

xHTTP — новый транспорт в Xray, имитирующий HTTP API-трафик.

**Конфигурация xhttpSettings в JSON (вставляется в Xray config через 3x-ui → Advanced):**

```json
"xhttpSettings": {
  "path": "/api/v1/update",
  "mode": "packet-up",
  "scMaxEachPostBytes": 1000000,
  "xPaddingBytes": "100-1000"
}
```

- `packet-up` — дробит трафик на мелкие POST-запросы, имитируя API
- `xPaddingBytes` — рандомный padding, усложняет статистический анализ

**Рекомендуется комбинировать:** VLESS + Reality + xHTTP транспорт.

### 7.3 Параметры клиента (важно)

В клиентских приложениях (v2rayN, Hiddify, Happ, v2raytun):

| Параметр | Значение |
|----------|----------|
| uTLS / Fingerprint | `chrome` |
| Flow | `xtls-rprx-vision` |
| SNI | тот же, что на сервере |
| ALPN | `h2` или `h2,http/1.1` |

### 7.4 DNS в панели 3x-ui

Обязательно настрой шифрованный DNS в разделе **Xray Settings → DNS**:

```json
{
  "servers": [
    {
      "address": "https://1.1.1.1/dns-query",
      "domains": ["geosite:geolocation-!cn"]
    },
    "8.8.8.8"
  ]
}
```

> Если браузер клиента шлёт DNS через системный резолвер провайдера — никакая маскировка не спасёт. DNS должен идти через туннель или быть зашифрован DoH/DoT.

### 7.5 Routing в 3x-ui

В разделе **Routing** добавь правила, чтобы российский трафик шёл напрямую (это уберёт подозрительный паттерн «всё идёт через один IP»):

```json
{
  "rules": [
    {
      "type": "field",
      "outboundTag": "direct",
      "domain": ["geosite:ru", "geosite:category-gov-ru"]
    },
    {
      "type": "field",
      "outboundTag": "direct",
      "ip": ["geoip:ru", "geoip:private"]
    }
  ]
}
```

## 8. Self-Steal и правильный SNI

**Главная проблема к концу 2025:** ТСПУ сравнивает IP из SNI (через DNS-резолв) с реальным IP назначения пакета. Если `SNI = ya.ru`, а пакет летит на `1.2.3.4` (хостинг в Финляндии) — разрыв.

### Self-Steal — правильная схема

1. Покупаешь домен (например, `example.com`)
2. Направляешь домен на IP своего VPS (A-запись)
3. На VPS поднимаешь веб-сервер (nginx) с реальной страницей-заглушкой
4. Получаешь TLS-сертификат (Let's Encrypt)
5. В 3x-ui: `Dest = example.com:443`, `SNI = example.com`

Теперь ТСПУ видит: SNI = `example.com`, IP = `1.2.3.4`. Резолвит `example.com` — получает `1.2.3.4`. Всё совпадает. Не блокирует.

**Схема nginx-заглушки:**

```nginx
server {
    listen 443 ssl http2;
    server_name example.com;
    
    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;
    
    # Реальный сайт-заглушка
    location / {
        root /var/www/html;
        index index.html;
    }
}
```

> В 3x-ui Reality работает поверх этого — реальные запросы к `example.com:443` обрабатывает nginx (видна легитимная страница), а VLESS-клиенты получают туннель прозрачно через Xray.

## 9. xHTTP — новый транспорт вместо голого Reality

Начиная с ноября 2025 года голый `VLESS+Reality` на TCP начинает детектироваться у МТС и МегаФон по поведенческим паттернам: DPI учится распознавать характерный для Reality ритм пакетов.

**xHTTP решает это** — трафик выглядит как серия HTTP POST-запросов к какому-то API.

**Как включить в 3x-ui:**

В настройках Inbound → Transport: выбрать `xhttp`, затем в поле Stream Settings JSON добавить:

```json
{
  "network": "xhttp",
  "xhttpSettings": {
    "path": "/api/v2/data",
    "mode": "packet-up",
    "scMaxEachPostBytes": 524288,
    "xPaddingBytes": "64-512"
  },
  "security": "reality",
  "realitySettings": {
    ...
  }
}
```

**Режимы xHTTP:**

| Режим | Описание | Подходит для |
|-------|----------|-------------|
| `packet-up` | POST-запросы вверх, GET-стрим вниз | Мобильные клиенты (лучший) |
| `stream-up` | Streaming upload | Десктоп |
| `stream-one` | Один поток | Тесты |

## 10. Каскадирование: схема «RU-прокладка → EU-сервер»

Самая надёжная схема для работы при любых белых списках.

```
[Клиент] → [RU-VPS в Яндекс.Облаке/VK Cloud] → [EU-VPS с 3x-ui]
```

### Реализация

**На RU-VPS (прокладка):**
- Устанавливаем простой SOCKS5 или Nginx stream proxy
- IP этого VPS находится в белом списке CIDR операторов
- Клиент подключается к нему без проблем

**На EU-VPS (основной):**
- Стоит 3x-ui с VLESS+Reality
- SNI настроен по схеме Self-Steal

**Конфиг Nginx stream на RU-прокладке:**

```nginx
stream {
    server {
        listen 443;
        proxy_pass eu-server-ip:443;
    }
}
```

Клиент в своём конфиге указывает адрес RU-прокладки, а SNI — домен EU-сервера. ТСПУ видит подключение к российскому IP → не блокирует. RU-VPS форвардит трафик дальше.

> ⚠️ **Важно:** не делай прокладку публичной. Используй IP-whitelist на nginx, чтобы только твои клиенты могли подключаться.

## 11. Podkop на роутере с OpenVPN

### Что такое Podkop

**Podkop** — open-source пакет для OpenWrt, реализующий умную точечную маршрутизацию. Под капотом — `sing-box` и технология FakeIP.

**Ключевая идея:** не весь трафик идёт в туннель, а только нужные домены и подсети. Остальное — напрямую. Банки не ругаются, пинг в играх нормальный, Netflix не видит VPN.

**Требования:**
- OpenWrt **24.10** (официальная, vanilla)
- Минимум 30 МБ свободной flash-памяти
- SSH доступ с root

### 11.1 Установка Podkop

```bash
sh <(wget -O - https://raw.githubusercontent.com/itdoginfo/podkop/refs/heads/main/install.sh)
```

Скрипт одновременно устанавливает и обновляет. После установки в LuCI появится раздел **Podkop**.

**Перед установкой удалить конфликтующие пакеты:**
- `getdomains` — конфликтует с Podkop
- `https-dns-proxy` — переписывает DNS-конфигурацию и ломает tproxy

### 11.2 Подключение OpenVPN-туннеля к Podkop

Podkop работает с OpenVPN в режиме **VPN-интерфейса** — Podkop не создаёт зону firewall и не делает перенаправление сам; это полностью на его стороне.

**Шаг 1: Установить OpenVPN**

```bash
opkg update
opkg install openvpn-openssl
```

Через LuCI: System → Software → `openvpn-openssl`.

**Шаг 2: Разместить конфиг**

Конфиг должен иметь расширение `.conf` (не `.ovpn`):

```bash
scp твой_конфиг.ovpn root@192.168.1.1:/etc/openvpn/client.conf
```

Или через LuCI в разделе OpenVPN.

**Типичная структура `.conf`:**

```
client
dev tun
proto udp          # или tcp, если UDP блокируется
resolv-retry infinite
nobind
persist-key
persist-tun
cipher AES-256-GCM
auth SHA512
verb 3
tls-client
tls-version-min 1.2
key-direction 1
remote-cert-tls server
dhcp-option DNS 1.1.1.1
dhcp-option DNS 1.0.0.1
remote 1.2.3.4 1194

<ca>
...
</ca>
<cert>
...
</cert>
<key>
...
</key>
<tls-auth>
...
</tls-auth>
```

> ⚠️ **Если UDP блокируется провайдером:** измени `proto udp` на `proto tcp`. Скорость упадёт, но соединение будет стабильнее.

**Шаг 3: Проверить, что туннель поднялся**

```bash
ip a | grep tun
traceroute itdog.info
```

В выводе `ip route` должна быть строка `0.0.0.0/1 via ... dev tun0`.

**Шаг 4: Настроить Podkop для работы с OpenVPN-интерфейсом**

В LuCI → Podkop:

1. **Connection Type:** VPN
2. **Interface:** выбрать `tun0` (имя интерфейса OpenVPN)
3. **Community Lists:** выбрать нужные списки (см. ниже)
4. Нажать **Save & Apply**

> Для OpenVPN-интерфейса можно задать отдельный **Domain Resolver** — DNS-сервер, запросы к которому будут идти через этот туннель. Например, `1.1.1.1` или `8.8.8.8`.

### 11.3 Секции Podkop: гибкая маршрутизация

Podkop поддерживает несколько секций — разный трафик в разные туннели.

**Пример конфигурации через UCI:**

```
config section 'main'
    option connection_type 'vpn'
    option interface 'tun0'
    list community_lists 'geoblock'
    list community_lists 'youtube'

config section 'direct_ru'
    option connection_type 'exclusion'
    list community_lists 'russia'
```

**Доступные community lists:**

| Список | Содержимое |
|--------|-----------|
| `russia_inside` | Все заблокированные в РФ ресурсы (Meta*, Twitter, Discord, Cloudflare) |
| `geoblock` | Геоблокированные для РФ сервисы |
| `youtube` | YouTube + связанные CDN |
| `telegram` | Telegram + подсети |
| `meta` | Instagram*, WhatsApp + подсети |
| `discord` | Discord + подсети |
| `cloudflare` | Подсети Cloudflare |

> Списки обновляются автоматически с GitHub: [itdoginfo/allow-domains](https://github.com/itdoginfo/allow-domains/)

### 11.4 DNS в Podkop

- **Домены из списков:** клиенты роутера получают FakeIP из диапазона `198.18.0.0/15`. Реальный IP резолвится на сервере (для VLESS/SS) или через указанный DNS-сервер в туннеле (для OpenVPN/WG).
- **Домены не из списков:** используется сервер из настроек → вкладка Settings → DNS-сервер.

**Рекомендуемый DNS для секции с OpenVPN:**

В настройках секции → **Domain Resolver:** `1.1.1.1` или `8.8.4.4`

Запросы к этому серверу пойдут через туннель `tun0`.

### 11.5 Проблема IPv6

Если отключил IPv6 на роутере и в логах OpenVPN видишь:

```
sitnl_send: rtnl: generic error (-13): Permission denied
Linux can't add IPv6 to interface tun0
```

Это нормально. Добавь в конфиг OpenVPN:

```
pull-filter ignore "ifconfig-ipv6"
pull-filter ignore "route-ipv6"
```

### 11.6 Проверка работы

```bash
# Смотрим таблицу маршрутов
ip route

# Проверяем трассировку
traceroute 8.8.8.8

# Логи OpenVPN
logread | grep openvpn

# Логи Podkop / sing-box
logread | grep sing-box
```

Если на первом хопе `traceroute` показывает не IP роутера, а чужой IP — трафик идёт через туннель.

### 11.7 Типовые схемы Podkop + OpenVPN

**Схема A: Всё заблокированное через OpenVPN**

```
Секция main:
  - Connection Type: VPN
  - Interface: tun0
  - Lists: russia_inside, youtube, telegram
```

**Схема B: YouTube через OpenVPN, остальное блокированное через VLESS**

```
Секция main (VLESS):
  - Connection Type: Proxy
  - Config: vless://...
  - Lists: geoblock, meta, discord, cloudflare

Секция youtube (OpenVPN):
  - Connection Type: VPN
  - Interface: tun0
  - Lists: youtube
```

**Схема C: Блокировка рекламных доменов**

```
Секция block_ads:
  - Connection Type: Block
  - Custom domain list: список рекламных доменов
```

Если nginx не установлен — установить.
Создать конфиг /etc/nginx/sites-enabled/[DOMAIN]:
- Слушать 80 и 443

    "network": "xhttp",
